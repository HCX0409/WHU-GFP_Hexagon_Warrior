【核心工程理念：存档驱动 (Checkpoint-Driven)】
本项目所有耗时操作（特征提取、模型训练、LoRA微调）均会产出“存档”（.joblib 文件或权重文件夹）。
后续流程（如序列打分）在运行时，会自动扫描本地目录：
- 若检测到存档：自动“读档”跳过耗时步骤，直接进行推理。
- 若未检测到存档：提示缺失，或自动触发前置计算（视具体命令而定）。
因此，队员间只需通过网盘/微信同步“存档文件”，即可在任何电脑上无缝接续运行后续流程。

【1. 目录与大文件管理规范 (极其重要！)】
绝对禁止将大文件直接 push 到 GitHub！所有数据、特征、模型存档必须通过团队网盘同步。
- `data/raw/`：原始数据 (GFP_data.xlsx, Exclusion_List.csv)。
- `data/processed/`：清洗后的中间数据 (full_sequences_full.csv)。
- `data/Feats/`：ESM-2 提取的特征存档 (.joblib)。
- `data/Models/`：
  - 传统模型存档 (.joblib)：如 XGBoost, Stacking 等。
  - 🌟 LoRA 微调存档 (文件夹)：如 ESM2_LoRA_CGH_...，包含 adapter_model 和 regressor.pt。这是微调的核心资产，必须完整同步！
- `generated_seqs/`：各阶段生成的候选序列 CSV。
- `models/`：预训练大模型本地缓存 (如 ESM-2 650M 权重)。

【2. 代码编写规范】
- 路径管理：绝对禁止硬编码绝对路径！必须使用 `src/config.py` 中的 `Path` 对象动态拼接。
- 类型与注释：核心函数必须包含 Type Hints 和 Google 风格的 Docstring。
- 随机种子：任何涉及随机性的操作，必须在函数开头调用 `config.seed_everything()`。

【3. Git 提交规范】
提交信息必须清晰，格式：`<type>: <subject>`
- `feat`: 新功能 (如: feat: 增加 ESM3 局部重绘模块)
- `fix`: 修复 bug (如: fix: 修复 1-based 索引越界问题)
- `refactor`: 代码重构 (如: refactor: 优化 PLL 计算逻辑)
- `docs`: 文档更新 (如: docs: 更新 README 运行指南)
